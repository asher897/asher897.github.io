Eliminating Synchronous Bottlenecks in Python Microservices

For years our worker processes handled one task at a time and it worked perfectly. We had enough logical separation and vertical scaling to handle the load. Once we started developing a new client-facing project with thousands of concurrent customers, blocking HTTP calls, database transactions, and file interactions made clear we were not primed for the expected load. Here's how I diagnosed it, what I built to fix it, and what I'd do differently.

Note: code and examples are illustrative — not representative of the actual proprietary implementation.

The Problem with "Good Enough"

The system had been running fine for years. Each microservice spun up a worker process that pulled messages off a RabbitMQ queue, did some work — called a downstream HTTP service, wrote to a database, published a result — then acknowledged the message and pulled the next one. Simple, predictable, easy to reason about.

The problem was in that last part: then pulled the next one. While the worker waited for an HTTP response or a database round-trip, it sat completely idle. Nothing else could run. The process was blocked. For a single worker handling light traffic, this was invisible. But scaling to a client-facing product with high concurrent load exposed it — those idle windows stacked up fast.

before — blocking worker loop 
def process_message(message: dict) -> None:
    # Blocks until HTTP response
    data = http_client.get(f"/api/records/{message['id']}")

    # Blocks until DB commit
    db.execute(
        "INSERT INTO results (id, payload) VALUES (?, ?)",
        (message['id'], data)
    )

    # Blocks until broker confirms
    channel.basic_publish(exchange='', routing_key='results', body=json.dumps(data))

def run():
    for method, props, body in channel.consume('tasks'):
        process_message(json.loads(body))
        channel.basic_ack(method.delivery_tag)

Every line touching a network resource parks the thread and burns wall-clock time. Under high load you can't scale out of it — you're just running more idle processes.

The key insight: these workers weren't CPU-bound. They were I/O-bound — almost all their time was spent waiting on network responses. That's exactly the problem async I/O was designed to solve.

Diagnosing the Bottleneck

Before writing a line of async code, I needed to be sure I was solving the right problem. Two things in our Grafana dashboards made the diagnosis unambiguous.

Queue depth grew non-linearly under load. Under moderate traffic, queues were roughly steady. Under burst load they grew faster than workers could drain them — not because workers were slow, but because each one was blocked on I/O while messages piled up behind it.

Downstream services were responding quickly. Individual HTTP calls and database queries were around sub-50ms each. But end-to-end processing time could reach 300–600ms. Those extra 250–500ms were pure wait time — sequential blocking calls that could have been running concurrently.

A word of caution: if your profiling shows CPU-bound workers — heavy data transformation, model inference, image processing — async won't help and may hurt. For CPU-bound work, reach for multiprocessing or offload to a dedicated compute service. Async is specifically a remedy for I/O-bound waiting.

The Async Worker Pattern

The core idea: instead of blocking the thread while waiting for I/O, yield control back to the event loop. It can then run other coroutines while the first one waits for its response. One thread, many concurrent tasks.

async worker — event loop entry 
import asyncio
import aioodbc
import aiohttp
from contextlib import asynccontextmanager

@asynccontextmanager
async def managed_resources():
    # I/O resources opened once at startup, shared across all coroutines
    async with aiohttp.ClientSession() as session:
        pool = await aioodbc.create_pool(dsn=DB_DSN, minsize=2, maxsize=10)
        try:
            yield session, pool
        finally:
            pool.close()
            await pool.wait_closed()

async def main():
    async with managed_resources() as (session, pool):
        await run_consumer(session, pool)

if __name__ == "__main__":
    asyncio.run(main())

The asynccontextmanager ensures the HTTP session and database connection pool are created once at startup, shared across all concurrent handlers, and cleanly torn down on shutdown. In the synchronous version, each handler was opening its own connections per message — an expensive habit that compounded the latency problem.

Bounding concurrency with the connection pool

Unlimited concurrent tasks can overwhelm downstream services or exhaust database connections. Rather than introducing a separate semaphore, I used the database connection pool itself as the concurrency ceiling. Every handler must acquire a connection before doing meaningful work, so maxsize on the pool naturally limits how many can be active simultaneously. One mechanism, one place to configure — this was ultimately the cleaner solution.

The RabbitMQ prefetch_count must be aligned to match. It controls how many unacknowledged messages the broker delivers to the consumer at once. If it exceeds maxsize, the broker floods the consumer faster than the pool can service — handlers pile up waiting for a connection and memory bloats:

pool-bounded consumer 
async def run_consumer(session: aiohttp.ClientSession, pool: aioodbc.Pool):
    tasks = set()

    async with get_rabbitmq_channel() as channel:
        # Broker delivers no more messages than the pool can service
        await channel.set_qos(prefetch_count=pool.maxsize)

        async for message in channel.consume("tasks"):
            task = asyncio.create_task(handle_message(message, session, pool))
            tasks.add(task)
            task.add_done_callback(tasks.discard)

The message handler

async message handler 
async def handle_message(message, session: aiohttp.ClientSession, pool: aioodbc.Pool) -> None:
    try:
        payload = json.loads(message.body)

        # HTTP fetch and audit log share no dependency — run concurrently
        data, _ = await asyncio.gather(
            fetch_upstream(session, payload['id']),
            log_received(pool, payload['id']),
        )

        # write_result depends on data — runs after gather completes
        await write_result(pool, payload['id'], data)
        await message.ack()

    except Exception as e:
        logger.error(f"Failed to process {payload.get('id')}: {e}")
        await message.nack(requeue=False)

asyncio.gather is where real concurrency materialises within a single handler. The HTTP fetch and the audit log write have no dependency on each other — they run simultaneously. Only write_result waits, because it needs the fetched data. In the synchronous version, all three ran sequentially.

Async Database Access with aioodbc

In the synchronous version, a single persistent connection was sufficient — only one message was in flight at a time. With concurrent handlers that falls apart immediately. Each handler needs its own connection, which means connection pooling is non-negotiable.

aioodbc wraps pyodbc in a thread pool to provide an async interface and supports connection pooling — the pool is created once and coroutines borrow connections from it rather than opening new ones per message:

async db with connection pool 
async def write_result(pool: aioodbc.Pool, record_id: str, data: dict) -> None:
    async with pool.acquire() as conn:
        async with conn.cursor() as cursor:
            await cursor.execute(
                "INSERT INTO results (id, payload, processed_at) VALUES (?, ?, GETUTCDATE())",
                (record_id, json.dumps(data))
            )
            await conn.commit()

async def log_received(pool: aioodbc.Pool, record_id: str) -> None:
    async with pool.acquire() as conn:
        async with conn.cursor() as cursor:
            await cursor.execute(
                "INSERT INTO audit_log (id, event) VALUES (?, 'received')",
                (record_id,)
            )
            await conn.commit()

pool.acquire() handles borrowing and return automatically — no connection leaks if an exception fires mid-handler. A handler that calls pool.acquire() when all connections are checked out simply waits until one is returned. This waiting behaviour is exactly what makes maxsize an effective concurrency limiter: handlers can't proceed past the pool acquisition until capacity is available.

Extras: Fire-and-Forget Background Tasks

This pattern wasn't used in the final production framework — the pool-bounded approach handled everything we needed — but it's worth covering for cases where some work genuinely doesn't need to block the main processing path: metrics emission, cache warming, non-critical notifications. The critical mistake is creating tasks without tracking them:

background task pattern — tracked, not lost 
class WorkerContext:
    def __init__(self):
        self._background_tasks: set[asyncio.Task] = set()

    def fire(self, coro) -> None:
        task = asyncio.create_task(coro)
        self._background_tasks.add(task)
        task.add_done_callback(self._on_task_done)

    def _on_task_done(self, task: asyncio.Task) -> None:
        self._background_tasks.discard(task)
        if not task.cancelled() and task.exception():
            logger.error(f"Background task failed: {task.exception()}")

    async def drain(self) -> None:
        if self._background_tasks:
            await asyncio.gather(*self._background_tasks, return_exceptions=True)

Without holding a reference, Python's garbage collector can destroy a task mid-execution. The set keeps tasks alive until they complete. drain() ensures they finish cleanly on shutdown rather than being silently dropped when the process exits.

Making It a Framework

We already had an existing worker framework in place. The goal wasn't to replace it — it was to extend it so teams could adopt async without rewriting their service logic. All the async machinery needed to disappear into a base class, leaving the developer interface unchanged: inherit, implement handle(), run.

base worker class 
class AsyncWorkerBase:
    def __init__(self, queue: str):
        self.queue = queue

    async def handle(self, payload: dict, session, pool) -> None:
        # Subclasses implement this — pure business logic only
        raise NotImplementedError

    async def run(self):
        async with managed_resources() as (session, pool):
            async with get_rabbitmq_channel() as channel:
                await channel.set_qos(prefetch_count=pool.maxsize)
                async for message in channel.consume(self.queue):
                    asyncio.create_task(self._safe_handle(message, session, pool))

    async def _safe_handle(self, message, session, pool):
        try:
            await self.handle(json.loads(message.body), session, pool)
            await message.ack()
        except Exception as e:
            logger.error(f"Handler error: {e}")
            await message.nack(requeue=False)


# A service implements handle() and nothing else
class RecordsWorker(AsyncWorkerBase):
    async def handle(self, payload: dict, session, pool) -> None:
        data, _ = await asyncio.gather(
            fetch_upstream(session, payload['id']),
            log_received(pool, payload['id']),
        )
        await write_result(pool, payload['id'], data)

if __name__ == "__main__":
    asyncio.run(RecordsWorker(queue="tasks").run())

Resource lifecycle, prefetch alignment, error handling, and graceful shutdown all live in the base class and never need to be thought about again. The first service migration took a day; subsequent ones took under an hour.

Key Takeaways

1. Profile before you rewrite. Async solves I/O-bound waiting. If your workers are CPU-bound, asyncio runs on a single thread and won't help — reach for multiprocessing instead.

2. Use the connection pool as your concurrency limiter. If every handler must acquire a DB connection to do real work, maxsize becomes a natural ceiling — no separate semaphore needed. Tune it carefully: too low leaves throughput on the table, too high and CPU spikes under load. Always align prefetch_count to match.

3. Never fire-and-forget without tracking. Untracked tasks are silently garbage collected mid-execution. Hold references in a set, log exceptions in the done callback, and drain on shutdown.

4. Open shared resources once, share them everywhere. Connection pools and HTTP sessions opened at startup and passed through context managers cost far less than per-message connection establishment.

5. Protect the developer interface. Teams shouldn't have to understand the concurrency model to use the framework. Absorb everything into a base class early — the interface stays familiar and adoption is fast.
