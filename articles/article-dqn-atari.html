Deep Q-Networks: Learning to Play Atari from Pixels

I implemented the Nature DQN from scratch — the architecture that first demonstrated an agent learning to play Atari games at human level directly from raw pixel input. What makes DQN interesting isn't just the result; it's the set of specific, motivated engineering choices that make the whole thing stable enough to train at all.

The Problem: Instability in Deep RL

Reinforcement learning has a clean conceptual foundation: an agent takes actions, receives rewards, and learns a policy that maximises cumulative return over time. The challenge is that combining this with deep neural networks introduces instabilities that make naive implementations fail almost immediately.

The core tension is that in supervised learning your training targets are fixed — a label is a label. In Q-learning, the target you're training toward is itself computed from your own network's current predictions. Update the network, and your targets shift. You're chasing a moving goalpost, and with a neural network doing the chasing, the feedback loops can become destructive quickly. Add to this that consecutive game frames are highly correlated — feeding them sequentially means the network sees near-identical inputs back to back, making gradient updates redundant and noisy.

The Nature DQN paper addressed both problems with two targeted mechanisms: experience replay and a frozen target network. The implementation I built uses both.

Q-Learning and Why It Needs Help

Q-learning is a value-based RL method. The agent learns a function Q(s, a) — the expected cumulative reward for taking action a in state s and following the optimal policy thereafter. Once you have a good Q-function, the policy is trivial: in any state, take the action with the highest Q-value.

The learning update is the temporal difference (TD) error — the difference between the Q-value you predicted and a bootstrapped estimate of what it should be, based on the reward you actually received plus the discounted value of the next state:

TD target and loss 
# The target: what the Q-value for (s, a) should have been
y_j = r + γ · max_a' Q_target(s', a')    # if not terminal
y_j = r                                   # if terminal (done)

# The loss: minimise the squared error between prediction and target
loss = MSE(Q(s, a), y_j)

The problem is Q_target. If it's the same network as Q, every gradient step changes both the prediction and the target simultaneously. The solution is to compute targets using a separate frozen copy of the network, updated periodically rather than every step.

Why freezing the target network matters: imagine trying to hit a moving target with each shot. If your aim adjusts the target's position too, the feedback becomes circular and unstable. Freezing the target network for a fixed number of steps gives the online network a stable objective to optimise toward, then snaps the target forward once the online network has improved.

The Architecture: CNN to Q-Values

The state input is not a single frame — it's a stack of four consecutive 84×84 greyscale frames. Stacking frames gives the network information about motion: it can infer the direction and speed of the ball from how its position changes across the four frames. A single frame is ambiguous; four frames tell a story.

Input: 4 × 84 × 84 (stacked greyscale frames) │ ▼ ┌─────────────────────────────────────────────────────┐ │ Conv2D(32 filters, 8×8 kernel, stride 4) + ReLU │ → 32 × 19 × 19 │ Conv2D(64 filters, 4×4 kernel, stride 2) + ReLU │ → 64 × 9 × 9 │ Conv2D(64 filters, 3×3 kernel, stride 1) + ReLU │ → 64 × 7 × 7 └─────────────────────────────────────────────────────┘ │ ▼ Flatten → 3136 │ ┌─────────────────────────────────────────────────────┐ │ Linear(3136 → 512) + ReLU │ │ Linear(512 → n_actions) │ ← Q-value per action └─────────────────────────────────────────────────────┘
Fig 1 — Nature DQN architecture. Three convolutional layers extract spatial features from the stacked frame input. Two fully connected layers map the flattened features to a Q-value for each possible action.

The convolutional layers handle the spatial feature extraction — learning to detect edges, the ball, the paddles, and their relative positions. The large initial kernel (8×8, stride 4) aggressively downsamples early, capturing coarse structure cheaply. Subsequent layers refine at finer resolution. The fully connected layers then map this spatial representation to Q-values over the action space.

DQN forward pass 
def forward(self, x):
    # x: (batch, 4, 84, 84) — stacked frames
    x = self.relu(self.conv1(x))   # → (batch, 32, 19, 19)
    x = self.relu(self.conv2(x))   # → (batch, 64,  9,  9)
    x = self.relu(self.conv3(x))   # → (batch, 64,  7,  7)
    x = x.view(x.size(0), -1)     # → (batch, 3136)
    x = self.relu(self.fc1(x))     # → (batch, 512)
    return self.fc2(x)             # → (batch, n_actions) — one Q-value per action

Experience Replay: Breaking Correlation

The second stability mechanism is experience replay. Rather than learning from transitions as they arrive — where each transition is highly correlated with the last — the agent stores every (state, action, reward, next_state, done) transition in a replay buffer and samples random minibatches from it during training.

This breaks the temporal correlation between consecutive training samples. A minibatch drawn randomly from 50,000 stored transitions will contain experiences from many different points in the game — early exploration, late exploitation, successful rallies, missed shots — giving the network a much more diverse and representative gradient signal.

TD loss computation with replay buffer 
def optimise_td_loss(self):
    batch = self.replay_buffer.sample(self.batch_size)

    state      = tensor(batch["state"],      dtype=float32)
    next_state = tensor(batch["next_state"], dtype=float32)
    reward     = tensor(batch["reward"],     dtype=float32)
    done       = tensor(batch["done"],       dtype=float32)
    action     = tensor(batch["action"])

    # Target: r  +  γ · max_a' Q_target(s', a')  ·  (1 - done)
    q_next = self.target_dqn(next_state).max(dim=1)[0]
    y_j = reward + (1 - done) * self.gamma * q_next

    # Prediction: Q(s, a) — only for the action actually taken
    q_pred = self.dqn(state).gather(1, action.unsqueeze(-1)).squeeze()

    loss = self.loss_fn(q_pred, y_j)
    loss.backward()
    self.optimizer.step()
    return loss

The .gather() call is worth noting: the network outputs a Q-value for every action, but we only want the Q-value for the action that was actually taken. gather indexes into the output tensor using the stored action indices, extracting exactly the predictions we want to supervise.

Exploration: ε-Greedy with Linear Decay

A purely greedy policy — always taking the action with the highest Q-value — will never discover better strategies it hasn't seen yet. The agent needs to explore. The standard approach is ε-greedy: with probability ε take a random action, otherwise act greedily. ε is annealed from 1.0 (fully random) down to 0.01 over the first 10% of training steps.

Starting fully random makes sense: the network's Q-values are meaningless at initialisation, so there's nothing to be greedy about. As the replay buffer fills with diverse experience and the network starts to learn, ε decays and the agent progressively trusts its own estimates more.

ε-greedy action selection 
fraction      = min(1.0, t / eps_timesteps)
eps_threshold = eps_start + fraction * (eps_end - eps_start)
# linearly anneals from 1.0 → 0.01 over eps_fraction of total steps

if random() < eps_threshold:
    action = env.action_space.sample()   # explore
else:
    action = argmax(Q_online(state))     # exploit

The Full Training Loop

The interactions between these components follow a specific sequence each step. The ordering matters — in particular, learning only begins after the replay buffer has accumulated enough transitions to sample meaningful minibatches from.

For each environment step t: 1. Select action via ε-greedy policy │ ▼ 2. Step environment → (next_state, reward, done) │ ▼ 3. Store (state, action, reward, next_state, done) in replay buffer │ ▼ 4. If t > learning_starts AND t % learning_freq == 0: └─ Sample minibatch from replay buffer └─ Compute TD targets using frozen target network └─ Backprop MSE loss through online network │ ▼ 5. If t > learning_starts AND t % target_update_freq == 0: └─ Hard copy online weights → target network
Fig 2 — Training loop structure. Learning is deferred until the buffer has enough transitions. Target network updates are less frequent than gradient steps, keeping targets stable between syncs.

The hyperparameters used in this implementation reflect the original Nature paper closely, with some reductions for the training budget available: replay buffer of 50k transitions (paper: 1M), batch size 64, learning rate 1e-4, discount factor γ=0.99, target network sync every 100 steps, training for 1M environment steps total.

Key Takeaways

1. The instabilities are real and the fixes are targeted. Without experience replay and a frozen target network, DQN training collapses. Each mechanism addresses a specific failure mode — correlated gradients and moving targets respectively. Understanding why each component exists matters more than knowing it exists.

2. Frame stacking is implicit temporal encoding. There's no recurrence in DQN — no LSTM, no explicit memory. The agent infers motion from the difference in object positions across four stacked frames. It's a simple and effective design choice that keeps the architecture tractable.

3. Delayed learning is intentional. Waiting until the replay buffer has 1,000+ transitions before starting gradient updates ensures the initial minibatches are diverse. Starting immediately would mean training on nearly identical transitions from the first few seconds of gameplay — exactly the correlation problem replay is meant to solve.

4. The CNN architecture encodes a strong prior. The choice of kernel sizes and strides (8×8/4, 4×4/2, 3×3/1) is deliberate — each layer operates at a different spatial scale, progressively building from low-level edges to higher-level game objects. This is not incidental; it reflects an understanding of how spatial hierarchies should be encoded for game-frame inputs.
