From SQL Transforms to Polars: Rethinking the ETL Layer

We were paying database compute costs to run transformations that had no business being in the database. Moving them to in-memory Polars — and wrapping the whole thing in a YAML-driven Airflow framework — cut costs, made pipelines testable, and reduced new pipeline creation to a config file. Here's how it was built.

The Problem with Transforming in the Database

The original setup was straightforward: raw client data lands in the database, a series of SQL queries run against it to clean, join, and reshape the data for actuarial analysis, and the results are written back out. It's a pattern that feels natural — SQL is expressive, the data is already there, so why move it?

The answer, at scale, is cost. Database compute is expensive. Every multi-table join, every aggregation over millions of rows, every regex substitution running inside the engine is burning money on provisioned capacity that sits idle the rest of the time. For ad-hoc queries this is fine. For systematic, repeatable transformation pipelines running on a schedule, it's the wrong tool.

There were three other compounding problems. Testability: SQL transformations living inside stored procedures or ad-hoc query files are difficult to unit test. You need a live database with the right schema and data to run them at all. Reproducibility: without strict version control and dependency tracking, it's easy to end up with transformations that work in production but can't be reliably re-run from scratch. Reusability: each new client dataset meant largely rewriting the same transformation logic with minor variations — no abstraction, no shared components, just more SQL.

The core shift: move the transformation layer out of the database and into Python, using Polars for in-memory processing. Load raw data from the database once, transform entirely in memory, write results back. The database becomes a dumb store — which is exactly what it should be in this pattern.

Why Polars over pandas

The natural Python choice for dataframe manipulation is pandas. But pandas has two properties that made it a poor fit here: it processes data eagerly row-by-row under the hood in ways that don't parallelise well, and its memory footprint at scale is large. For transformation pipelines over client datasets that could run into millions of rows, these matter.

Polars is built on Apache Arrow and uses a lazy query engine backed by Rust. The practical implications are meaningful: operations are vectorised across columns, parallelism happens automatically across CPU cores, and the lazy API allows the query planner to optimise the full transformation graph before executing any of it — pushing filters early, eliminating redundant passes over data, merging operations where possible. For the kinds of transformations in these pipelines — large joins, grouped aggregations, string cleaning, window functions — Polars consistently outperforms pandas by a significant margin and uses substantially less memory.

There's also an expressiveness argument. Polars' method-chaining API maps cleanly onto transformation logic — a sequence of operations reads like a description of the pipeline, which matters when the person reading it next isn't the person who wrote it.

The Transformations in Practice

To make the comparison concrete, here are the kinds of transformations that moved from SQL to Polars.

Joins and aggregations

before — SQL in the database 
SELECT
    p.patient_id,
    p.admission_date,
    p.discharge_date,
    DATEDIFF(day, p.admission_date, p.discharge_date) AS los_days,
    COUNT(c.claim_id)                                  AS claim_count,
    SUM(c.claim_amount)                                AS total_claimed
FROM patients p
LEFT JOIN claims c
    ON p.patient_id = c.patient_id
    AND c.claim_date BETWEEN p.admission_date AND p.discharge_date
GROUP BY
    p.patient_id, p.admission_date, p.discharge_date
after — polars in-memory 
import polars as pl

patients = pl.scan_csv("patients.csv")   # lazy — nothing loaded yet
claims   = pl.scan_csv("claims.csv")

result = (
    patients
    .join(claims, on="patient_id", how="left")
    .filter(
        pl.col("claim_date").is_between(
            pl.col("admission_date"), pl.col("discharge_date")
        )
    )
    .group_by(["patient_id", "admission_date", "discharge_date"])
    .agg([
        pl.col("claim_id").count().alias("claim_count"),
        pl.col("claim_amount").sum().alias("total_claimed"),
        (pl.col("discharge_date") - pl.col("admission_date"))
          .dt.total_days().first().alias("los_days"),
    ])
    .collect()   # execution happens here — query plan optimised first
)

The Transform / Transformer Framework

Replacing SQL with Polars solves the compute cost problem. But a collection of standalone Polars scripts is still just a collection of standalone scripts — hard to compose, hard to test in isolation, and requiring bespoke wiring for every new pipeline. The goal was a proper abstraction layer: define transformation logic once as reusable classes, declare pipelines in config, and let a builder assemble and run them.

The solution is a two-level class hierarchy. Transform encapsulates a single operation on a DataFrame. Transformer holds an ordered group of Transforms and runs them in sequence. A pipeline's YAML config declares which Transformers to build and which predefined Transforms go inside each one — the builder reads it, instantiates the objects, and wires them into Airflow tasks.

The Transform class

A Transform knows its input column names, its output column names, and optionally any dimension DataFrames it needs — lookup tables, reference data, anything that isn't the primary input. Its only public method is apply(): takes a DataFrame, returns a DataFrame.

transform base class 
from dataclasses import dataclass, field
from abc import ABC, abstractmethod
import polars as pl

@dataclass
class Transform(ABC):
    input_cols:  list[str]
    output_cols: list[str]
    dimensions:  dict[str, pl.DataFrame] = field(default_factory=dict)

    @abstractmethod
    def apply(self, df: pl.DataFrame) -> pl.DataFrame: ...

Concrete subclasses implement apply() with specific Polars logic. The dimensions dict keeps apply()'s signature clean regardless of complexity — a join against a large reference table and a simple column rename both have the same interface. input_cols and output_cols can be validated at build time, catching schema mismatches before a DAG run fails mid-execution.

predefined transform subclasses 
@dataclass
class NormaliseCodesTransform(Transform):
    def apply(self, df: pl.DataFrame) -> pl.DataFrame:
        return df.with_columns([
            pl.col(c).str.strip_chars().str.to_uppercase()
            for c in self.input_cols
        ])

@dataclass
class JoinDimensionTransform(Transform):
    on: str = ""
    def apply(self, df: pl.DataFrame) -> pl.DataFrame:
        return df.join(self.dimensions["dim"], on=self.on, how="left")

@dataclass
class RollingWindowTransform(Transform):
    partition_by: str = ""
    order_by:     str = ""
    window_days:  int = 90
    def apply(self, df: pl.DataFrame) -> pl.DataFrame:
        return (
            df.sort([self.partition_by, self.order_by])
            .with_columns([
                pl.col(c)
                  .rolling_sum(window_size=self.window_days,
                               by=self.order_by, closed="right")
                  .over(self.partition_by)
                  .alias(f"{c}_rolling_{self.window_days}d")
                for c in self.input_cols
            ])
        )

The Transformer class

A Transformer is a deliberately thin ordered container. It calls each Transform's apply() in sequence, piping the output of one into the input of the next:

transformer — ordered composition 
@dataclass
class Transformer:
    name:       str
    transforms: list[Transform]

    def run(self, df: pl.DataFrame) -> pl.DataFrame:
        for t in self.transforms:
            df = t.apply(df)
        return df

Declaring pipelines in YAML

Each pipeline is declared in a YAML file. The config lists Transformers by name, and each Transformer lists its Transforms by name — each with the column and parameter config the builder needs to instantiate them. The builder reads this, looks up each Transform class by name in a registry, instantiates it with the provided config, and assembles the Transformer:

pipeline config — transformers and transforms declared in yaml 
pipeline:
  name: patient_episode_summary
  schedule: "@daily"

  sources:
    patients:
      type: sql
      query: SELECT * FROM patients WHERE active = 1
      connection: client_db
    dim_icd:
      type: sql
      query: SELECT code, description FROM icd_reference
      connection: client_db

  transformers:
    - name: cleaning
      transforms:
        - type: NormaliseCodesTransform
          input_cols:  [diagnosis_code, procedure_code]
          output_cols: [diagnosis_code, procedure_code]

        - type: JoinDimensionTransform
          input_cols:  [diagnosis_code]
          output_cols: [diagnosis_code, icd_description]
          dimension_source: dim_icd
          on: diagnosis_code

    - name: enrichment
      transforms:
        - type: RollingWindowTransform
          input_cols:  [claim_amount]
          output_cols: [claim_amount_rolling_90d]
          partition_by: patient_id
          order_by: episode_date
          window_days: 90

  output:
    destination: client_db
    table: episode_summary
    mode: replace

The builder

The builder reads the config, instantiates each declared Transform from the registry, assembles the Transformers, and wires each one into an Airflow task. Adding a new pipeline is a YAML file — no builder changes, no new Python wiring:

builder — yaml to transformer instances to airflow tasks 
import yaml, importlib, glob, os
from airflow import DAG
from airflow.operators.python import PythonOperator
from datetime import datetime

# Registry maps transform type name → class
TRANSFORM_REGISTRY: dict[str, type[Transform]] = {
    "NormaliseCodesTransform": NormaliseCodesTransform,
    "JoinDimensionTransform":  JoinDimensionTransform,
    "RollingWindowTransform":  RollingWindowTransform,
}

def build_transform(cfg: dict, sources: dict) -> Transform:
    cls = TRANSFORM_REGISTRY[cfg["type"]]
    dims = {}
    if "dimension_source" in cfg:
        dims["dim"] = sources[cfg["dimension_source"]]
    return cls(
        input_cols=cfg["input_cols"],
        output_cols=cfg["output_cols"],
        dimensions=dims,
        **{k: v for k, v in cfg.items()
           if k not in {"type", "input_cols", "output_cols", "dimension_source"}},
    )

def build_dag(config_path: str, sources: dict) -> DAG:
    with open(config_path) as f:
        cfg = yaml.safe_load(f)["pipeline"]

    transformers = [
        Transformer(
            name=t_cfg["name"],
            transforms=[build_transform(tr, sources) for tr in t_cfg["transforms"]],
        )
        for t_cfg in cfg["transformers"]
    ]

    dag = DAG(cfg["name"], start_date=datetime(2024, 1, 1),
              schedule_interval=cfg.get("schedule", "@daily"), catchup=False)

    prev_task = None
    with dag:
        for transformer in transformers:
            task = PythonOperator(
                task_id=transformer.name,
                python_callable=transformer.run,
            )
            if prev_task:
                prev_task >> task
            prev_task = task
    return dag
Testing the hierarchy: because apply() takes a plain Polars DataFrame and returns one, unit testing a Transform requires no Airflow, no database, no infrastructure — just a DataFrame with the right schema and an assertion on the result. A Transformer test calls run() and asserts the final output. The entire transformation layer is covered without spinning up a single external dependency.

Wiring It into Airflow

Airflow discovers DAGs by scanning a directory for Python files that expose DAG objects at module level. A thin loader script does nothing but find YAML configs and call the builder — the builder handles everything from there:

airflow dag loader — auto-discovers yaml configs 
import os, glob
from sources import load_sources   # loads declared SQL sources from config

CONFIG_DIR = os.environ.get("PIPELINE_CONFIG_DIR", "/opt/pipelines/configs")

for config_path in glob.glob(f"{CONFIG_DIR}/*.yaml"):
    sources  = load_sources(config_path)
    dag_name = os.path.splitext(os.path.basename(config_path))[0]
    globals()[dag_name] = build_dag(config_path, sources)

Adding a new pipeline means one YAML file. Airflow picks it up on the next scan, the builder instantiates the declared Transforms and Transformers, and the DAG appears with one task per Transformer — named, ordered, and independently retryable.

What Actually Changed

Database compute cost dropped significantly. The database now handles only raw data storage and simple SELECT queries to extract source data. All transformation logic — joins, aggregations, string cleaning, window functions — runs in-memory on the Airflow worker. For large client datasets, the cost difference per pipeline run is meaningful, and it compounds across every client and every scheduled run.

Pipeline creation time fell from days to hours. Before the framework, a new client pipeline meant writing SQL transforms, setting up database objects, and manually wiring Airflow tasks. After, it meant writing a YAML config against the existing transform registry. Standard operations — joins, aggregations, cleaning — need no new code at all.

Transforms became testable. A transform function takes a Polars DataFrame and returns a Polars DataFrame. Testing it requires no database, no Airflow, no infrastructure — just a DataFrame with the right schema and a pytest assertion. This is the biggest quality-of-life improvement: catching transform logic errors in a test run rather than a failed Airflow DAG.

Pipelines became auditable. The YAML config is version-controlled. The Airflow DAG run logs are structured and searchable. Every pipeline run is reproducible from the same config against the same source data. This matters for actuarial analysis where the provenance of a data preparation step can be questioned.

Where this pattern breaks down: in-memory processing has a memory ceiling. For datasets large enough that they don't fit comfortably in the worker's RAM, Polars' lazy streaming mode helps — but there's a point where you're better off pushing the aggregation back to the database or moving to a distributed compute layer like Spark. Know your data sizes. For the datasets in this context — client health insurance data, tens of millions of rows at most — in-memory Polars was well within bounds.

Key Takeaways

1. The database is not a compute engine for your application logic. Using it for systematic data transformation is convenient until it isn't — then it's expensive, opaque, and hard to test. Move transformation logic to where it belongs: application code, with version control, unit tests, and a proper abstraction layer.

2. Polars' lazy API is the right default. Build the full transformation chain as a lazy query, let the planner optimise it, then call .collect() once at the end. Calling .collect() mid-chain — to inspect an intermediate result — breaks the optimisation. Use .fetch(n) for development inspection instead.

3. Separate transform logic from pipeline wiring. Transforms are pure data operations; Transformers are composition containers; the DAG builder is orchestration. Keeping these three concerns separate means each can be changed, tested, and extended independently. Adding a new Transform type doesn't touch the DAG builder; restructuring Transformer groups doesn't touch the Transform classes.

4. Validate schemas at construction time. Because each Transform declares its input_cols and output_cols, the full pipeline's schema contract can be validated when the DAG is built — before any data is loaded. Catching a missing column at construction time is far cheaper than catching it in a running Airflow task an hour into a pipeline run.

5. Transformer boundaries are natural Airflow task boundaries. Grouping Transforms into Transformers isn't just about organisation — it maps directly to Airflow task retries, logging, and monitoring. When a pipeline fails, the failed Transformer name tells you exactly which logical group of operations broke. You can retry just that task without re-running the whole pipeline.
